## 概述

點餐系統的訂單狀態流程依業態（外帶、堂食、叫號）有不同設計。
核心原則：**每個狀態對應一個實際動作**，避免模糊的中間狀態。

---

## Option A：最簡流程

適合：小規模外帶、飲料店、自助點餐機

```
待處理 (pending)
  │
  ├─[處理]→ 處理中 (processing)
  │             │
  │             └─[完成]→ 已完成 (completed)
  │
  └─[任何階段]→ 已取消 (cancelled)
```

**狀態定義**

| 狀態 | 觸發者 | 說明 |
|---|---|---|
| `pending` | 顧客送出 | 訂單進入隊列 |
| `processing` | 員工點擊「處理」 | 正在備餐 |
| `completed` | 員工點擊「完成」 | 結案 |
| `cancelled` | 員工 | 任何階段取消 |

✅ 實作簡單，介面操作少
❌ 無法區分「備好了」與「已取走」

---

## Option B：加入可取餐狀態（推薦）

適合：咖啡廳、快餐、小型餐廳、自取式服務

```
待處理 (pending)
  │
  ├─[開始備餐]→ 備餐中 (preparing)
  │                 │
  │                 └─[備好了]→ 可取餐 (ready)
  │                                 │
  │                                 └─[已取餐]→ 已完成 (completed)
  │
  └─[任何階段]→ 已取消 (cancelled)
```

**狀態定義**

| 狀態 | 觸發者 | 說明 |
|---|---|---|
| `pending` | 顧客送出 | 訂單進入廚房隊列，等待處理 |
| `preparing` | 廚房員工 | 開始製作，廚房顯示卡片變色 |
| `ready` | 廚房員工 | 備餐完成，通知外場出餐或叫號 |
| `completed` | 外場員工 / 顧客 | 已取餐 / 已送達，結案 |
| `cancelled` | 員工 | 任何階段皆可取消 |

✅ 每個狀態對應明確動作
✅ 可延伸叫號、螢幕通知、手機推播
✅ 報表可計算「備餐等待時間」與「取餐等待時間」
❌ 多一個 `ready → completed` 步驟，小店可能覺得繁瑣

---

## Option C：堂食完整流程

適合：正式餐廳、需要接單確認、外送平台整合

```
待確認 (pending)
  │
  ├─[接單]→ 備餐中 (preparing)
  │             │
  │             └─[出餐]→ 已出餐 (served)
  │                           │
  │                           └─[結帳離桌]→ 用餐完畢 (completed)
  │
  └─[拒絕]→ 已拒絕 (rejected)   ←（任何階段也可取消）
```

**狀態定義**

| 狀態 | 觸發者 | 說明 |
|---|---|---|
| `pending` | 顧客送出 | 等待店家確認是否接單 |
| `preparing` | 店家接單 | 廚房開始製作 |
| `served` | 外場員工 | 菜送至桌上 |
| `completed` | 系統 / 員工 | 顧客結帳離桌，結案 |
| `rejected` | 店家 | 無法接單（客滿、缺料等） |
| `cancelled` | 員工 | 顧客取消 |

✅ 完整追蹤堂食服務流程
✅ 適合整合外送平台接單邏輯
❌ 步驟多，小店日常操作繁瑣

---

## Option D：外帶叫號流程

適合：有叫號機 / 螢幕叫號系統的快餐店、美食廣場

```
待處理 (pending)
  │
  ├─[開始]→ 備餐中 (preparing)
  │             │
  │             └─[叫號]→ 待取餐 (calling)   ← 螢幕顯示號碼
  │                           │
  │                           └─[取餐完成]→ 已完成 (completed)
  │
  └─[任何階段]→ 已取消 (cancelled)
```

**狀態定義**

| 狀態 | 觸發者 | 說明 |
|---|---|---|
| `pending` | 顧客 | 取號排隊 |
| `preparing` | 員工 | 開始製作 |
| `calling` | 員工 | 叫號中，顯示於叫號螢幕 |
| `completed` | 顧客取餐後員工確認 | 結案 |
| `cancelled` | 員工 | 取消 |

✅ 叫號狀態驅動螢幕顯示
✅ 清楚區分「備好了但未取」
❌ 需配合叫號前端實作

## 選擇建議

| 業態 | 建議流程 |
|---|---|
| 飲料店、小吃、自助點餐 | Option A |
| 咖啡廳、輕食、小型餐廳 | **Option B（推薦）** |
| 正式餐廳、外送平台整合 | Option C |
| 美食廣場、叫號快餐 | Option D |

> **延伸性考量**：Option B 最容易延伸至叫號（`ready` 狀態觸發螢幕顯示）
> 或推播通知（`ready` 時發送 LINE / SMS），是彈性最高的起點。
