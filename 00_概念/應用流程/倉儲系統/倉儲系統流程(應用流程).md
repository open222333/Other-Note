## 概述

倉儲系統的核心流程包含：**入庫、出庫、調撥、盤點、退貨、庫存預警**。
每個流程依規模與管理複雜度有不同設計。

核心原則：
- 每筆庫存異動都有對應單據可追溯
- 單據狀態驅動庫存數字，不允許直接修改庫存
- 草稿階段可自由編輯，確認後不可逆

---

## 一、入庫流程

### Option A：簡易入庫

適合：小型倉庫、個人管理、快速補貨

```
建立入庫記錄
     │
     └─[確認]→ 庫存增加 → 完成
               (同時建立移動紀錄)
```

**步驟說明**

| 步驟 | 操作者 | 動作 |
|---|---|---|
| 建立記錄 | 倉管 | 選產品、倉庫、數量 |
| 確認入庫 | 倉管 | 庫存立即增加，不可撤銷 |

✅ 操作最少，速度快
❌ 無採購依據，無法對帳

---

### Option B：採購單驅動入庫（推薦）

適合：中小型企業、需要採購對帳

```
建立採購單 (draft)
     │
     ├─[編輯品項]→ 草稿 (draft)
     │
     ├─[確認採購]→ 待入庫 (confirmed)
     │                 │
     │                 └─[實際收貨]→ 已入庫 (completed)
     │                               (庫存增加，建立移動紀錄)
     │
     └─[任何階段]→ 已取消 (cancelled)
```

**狀態定義**

| 狀態 | 觸發者 | 庫存異動 | 說明 |
|---|---|:---:|---|
| `draft` | 採購人員 | ❌ | 可自由增刪品項、修改數量 |
| `confirmed` | 主管 / 系統 | ❌ | 採購金額鎖定，等待到貨 |
| `completed` | 倉管收貨 | ✅ 增加 | 實際入庫，庫存更新 |
| `cancelled` | 採購人員 | ❌ | 作廢，不影響庫存 |

✅ 採購與入庫可分工
✅ 可對帳（採購金額 vs 實際到貨）
✅ `confirmed` 狀態可做庫存預計到貨量

---

### Option C：多步驟品質管控入庫

適合：製造業、需要驗貨 / 品管的倉庫

```
建立採購單 (draft)
     │
     ├─[送審]→ 待審核 (pending_approval)
     │              │
     │         [核准]→ 待到貨 (confirmed)
     │                    │
     │               [到貨登記]→ 待驗貨 (receiving)
     │                               │
     │                     [驗貨通過]→ 已入庫 (completed)
     │                     [驗貨不通過]→ 退回廠商 (rejected)
     │
     └─[任何草稿 / 審核階段]→ 已取消 (cancelled)
```

**狀態定義**

| 狀態 | 說明 |
|---|---|
| `draft` | 草稿，採購人員編輯中 |
| `pending_approval` | 送主管審核，等待核准 |
| `confirmed` | 核准，通知廠商出貨 |
| `receiving` | 貨品到倉，等待驗貨 |
| `completed` | 驗貨通過，庫存增加 |
| `rejected` | 驗貨不通過，退回廠商 |
| `cancelled` | 作廢 |

✅ 完整稽核追蹤
✅ 品管驗貨有明確節點
❌ 流程長，小型倉庫操作繁瑣

## 二、出庫流程

### Option A：簡易出庫

適合：小型倉庫、內部領料

```
建立出庫記錄
     │
     └─[確認]→ 庫存減少 → 完成
```

✅ 最快速
❌ 無銷售依據，無法對帳

---

### Option B：銷售單驅動出庫（推薦）

適合：中小型企業、銷售與倉庫分工

```
建立出庫單 (draft)
     │
     ├─[編輯品項]→ 草稿 (draft)
     │
     ├─[確認出庫]→ 已出庫 (completed)
     │              (庫存減少，建立移動紀錄)
     │
     └─[草稿階段]→ 已取消 (cancelled)
```

**狀態定義**

| 狀態 | 觸發者 | 庫存異動 | 說明 |
|---|---|:---:|---|
| `draft` | 業務 / 倉管 | ❌ | 可自由修改 |
| `completed` | 倉管出貨 | ✅ 減少 | 實際出庫，庫存更新 |
| `cancelled` | 業務 / 倉管 | ❌ | 作廢 |

---

### Option C：撿貨驗貨出庫

適合：電商倉、大型倉庫、多品項訂單

```
建立出庫單 (draft)
     │
     ├─[排程]→ 待撿貨 (picking)
     │              │
     │         [撿貨完成]→ 待驗貨/包裝 (packing)
     │                         │
     │                    [出貨]→ 已出庫 (shipped)
     │                               │
     │                          [簽收]→ 已送達 (delivered)
     │
     └─[任何階段]→ 已取消 (cancelled)
```

**狀態定義**

| 狀態 | 說明 |
|---|---|
| `draft` | 草稿 |
| `picking` | 倉庫人員撿貨中 |
| `packing` | 驗貨、打包中 |
| `shipped` | 已交物流，庫存減少 |
| `delivered` | 客戶簽收確認 |
| `cancelled` | 作廢 |

✅ 完整電商出貨流程
✅ 可接物流 API 追蹤
❌ 流程長，小型倉庫過重

---

## 三、調撥流程

### Option A：直接調撥

適合：同一管理者管理的多個倉庫

```
建立調撥申請
     │
     └─[確認]→ 來源倉庫庫存減少
               目標倉庫庫存增加
               (同時建立兩筆移動紀錄: transfer_out / transfer_in)
```

✅ 最簡單，一步完成
❌ 無法處理「在途庫存」（貨在路上）

---

### Option B：調撥單審核流程（推薦）

適合：多倉庫管理、異地倉、需要在途庫存追蹤

```
建立調撥單 (draft)
     │
     ├─[送出]→ 待出庫 (pending)
     │              │
     │         [來源倉確認出庫]→ 在途 (in_transit)
     │              (來源倉庫存減少)  │
     │                          [目標倉確認收貨]→ 已完成 (completed)
     │                               (目標倉庫存增加)
     │
     └─[在完成前]→ 已取消 (cancelled)
                   (若已出庫則回補來源倉庫存)
```

**狀態定義**

| 狀態 | 庫存異動 | 說明 |
|---|:---:|---|
| `draft` | ❌ | 草稿，可修改 |
| `pending` | ❌ | 等待來源倉確認出庫 |
| `in_transit` | 來源倉 ✅ 減少 | 貨品在途，目標倉尚未入帳 |
| `completed` | 目標倉 ✅ 增加 | 雙邊確認，調撥完成 |
| `cancelled` | 視階段回補 | 作廢 |

✅ 支援在途庫存追蹤
✅ 雙倉各自確認，責任清晰
✅ 若貨品遺失可從 `in_transit` 查出

## 四、盤點流程

### Option A：直接盤點修正

適合：小型倉庫、快速修正

```
查詢現有庫存
     │
     └─[輸入實際數量]→ 系統計算差異
                            │
                       [確認]→ 庫存更新
                               建立「盤點調整」移動紀錄
```

✅ 最快速，適合日常微調
❌ 無盤點任務管理，難以追蹤執行進度

---

### Option B：盤點任務流程（推薦）

適合：中型倉庫、定期全盤或分區盤點

```
建立盤點任務 (created)
     │
     ├─[開始盤點]→ 盤點中 (counting)
     │                 │
     │            [人員填寫實際數量]
     │                 │
     │            [提交盤點]→ 待審核 (reviewing)
     │                            │
     │                   [審核通過]→ 執行調整 → 已完成 (completed)
     │                   [審核退回]→ 盤點中 (counting)  ← 重新盤點
     │
     └─[盤點中以前]→ 已取消 (cancelled)
```

**狀態定義**

| 狀態 | 說明 |
|---|---|
| `created` | 任務已建立，尚未開始 |
| `counting` | 人員盤點中，填寫實際數量 |
| `reviewing` | 提交給主管審核差異 |
| `completed` | 審核通過，庫存差異已調整 |
| `cancelled` | 取消盤點任務 |

✅ 盤點流程有紀錄可查
✅ 差異需審核才更新，避免誤操作
✅ 可分配給不同人員負責不同區域

---

### Option C：循環盤點（大型倉庫）

適合：大型倉庫、商品種類多、無法停工全盤

```
系統依規則自動產生每日盤點任務
（依 ABC 分類：A 類每週、B 類每月、C 類每季）
     │
     └─ 同 Option B 盤點任務流程
```

**ABC 分類原則**

| 類別 | 定義 | 盤點頻率 |
|---|---|---|
| A 類 | 高價值 / 高周轉率，佔庫存價值 70~80% | 每週 |
| B 類 | 中等價值 / 周轉率，佔 15~25% | 每月 |
| C 類 | 低價值 / 低周轉率，佔 5~10% | 每季 |

✅ 不需停工，分批執行
✅ 高價值商品頻繁確認，降低損失風險
❌ 需要系統支援自動排程與 ABC 分類管理

---

## 五、退貨流程

### Option A：直接退貨入庫

適合：小型倉庫、退貨品項少

```
收到退貨
     │
     └─[確認]→ 庫存增加（若良品）
               或建立報廢記錄（若瑕疵品）
```

---

### Option B：退貨審核流程

適合：需要品管確認、區分良品 / 瑕疵品

```
建立退貨單 (created)
     │
     ├─[收到貨品]→ 待檢驗 (inspecting)
     │                 │
     │         [良品]→ 入庫 (restocked)      ← 庫存增加
     │         [瑕疵品]→ 待處理 (defective)
     │                       │
     │              [報廢]→ 已報廢 (scrapped)
     │              [退廠商]→ 已退回 (returned_to_supplier)
     │              [維修後入庫]→ 入庫 (restocked)
     │
     └─[任何階段]→ 已取消 (cancelled)
```

**狀態定義**

| 狀態 | 說明 |
|---|---|
| `created` | 退貨單建立 |
| `inspecting` | 貨品到倉，等待品管 |
| `restocked` | 良品，已回補庫存 |
| `defective` | 瑕疵品，等待後續處理 |
| `scrapped` | 報廢處理 |
| `returned_to_supplier` | 退回供應商 |
| `cancelled` | 作廢 |

✅ 完整追蹤退貨品質狀況
✅ 瑕疵品可多種後續處置
❌ 流程較長

---

## 六、庫存預警流程

```
定期或即時檢查庫存
     │
     ├─[低於最低庫存]→ 觸發「低庫存警示」
     │                      │
     │               [自動 / 手動]→ 建立補貨採購單
     │
     └─[高於最高庫存]→ 觸發「超量警示」
                            │
                    [自動 / 手動]→ 建立調撥或促銷處理
```

**預警類型**

| 類型 | 觸發條件 | 建議動作 |
|---|---|---|
| 低庫存 | `quantity < min_stock`（且 `min_stock > 0`）| 建立補貨採購單 |
| 超量 | `quantity > max_stock`（且 `max_stock > 0`）| 調撥至其他倉或促銷 |
| 即將到期 | `expiry_date < today + N 天` | 優先出庫或促銷 |
| 長期滯銷 | 超過 N 天無出庫紀錄 | 盤點確認或促銷 |

## 單據狀態通用對照表

| 狀態 key | 中文 | 庫存異動 | 適用單據 |
|---|---|:---:|---|
| `draft` | 草稿 | ❌ | 入庫、出庫、調撥 |
| `pending` / `confirmed` | 待處理 / 已確認 | ❌ | 入庫、調撥 |
| `pending_approval` | 待審核 | ❌ | 入庫（品管流程）|
| `receiving` | 待驗貨 | ❌ | 入庫（品管流程）|
| `picking` | 撿貨中 | ❌ | 出庫（電商）|
| `packing` | 包裝中 | ❌ | 出庫（電商）|
| `in_transit` | 在途 | 來源倉 ✅ | 調撥 |
| `counting` | 盤點中 | ❌ | 盤點 |
| `reviewing` | 審核中 | ❌ | 盤點 |
| `completed` | 已完成 | ✅ | 全部 |
| `cancelled` | 已取消 | ❌（視情況回補）| 全部 |
| `rejected` | 已拒絕 / 退回 | ❌ | 入庫（品管）、退貨 |

## 選擇建議

| 規模 / 業態 | 入庫 | 出庫 | 調撥 | 盤點 |
|---|---|---|---|---|
| 個人 / 微型倉庫 | Option A | Option A | Option A | Option A |
| 中小型企業倉庫 | **Option B** | **Option B** | **Option B** | **Option B** |
| 製造業 / 品管需求 | Option C | Option C | Option B | Option B/C |
| 電商倉 / 大型倉庫 | Option B/C | Option C | Option B | Option C |

> **核心建議**：
> - 無論哪種流程，**所有庫存異動必須有對應單據**，禁止直接修改庫存數字
> - 單據從「草稿」到「確認」應為**不可逆**操作，需要有取消補救機制
> - 移動紀錄（`stock_movements`）獨立保存，作為稽核日誌，**不可刪除**
