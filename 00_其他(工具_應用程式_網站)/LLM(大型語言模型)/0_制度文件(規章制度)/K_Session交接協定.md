# 文件 K｜Session 交接協定（SESSION_HANDOFF）

> 版本：1.0（2026-07-07）
> 定位：對治兩個已確認限制——`/compact` 後召回衰退（文件 J 第 5 章）、以及手機 App 看不到 Claude.ai 上設定的 skill。把 G 與 skill 的關鍵片段濃縮成一份可貼上的最小上下文。
> 執行方式：每次開新 session／跨裝置接續時，先貼「開場快照」（第 1 章）；判斷是否該換 session 依第 3 章決策樹。

---

## 第 1 章：開場快照（固定欄位，開新 session 第一則訊息）

貼上以下模板並填滿；空欄以 `<待填>` 標記，不得留空跳過：

```
【角色】S / W / R（本次擔任）
【制度引用】遵守 A~M（衝突時：F > C；token 事宜以 J 為準）
【環境指標】引用 G v<版本>；易腐項是否已核對：chat_id / host key / GPG / 認證 → <是/否>
【進行中任務】<一句話目標>
【上次結論】<上個 session 的關鍵決定，或「無，全新任務」>
【回滾狀態】<若有未完成的正式環境變更：貼回滾指令；否則「無」>
【只准讀】<本次允許讀取的檔案清單，來自 D>
【本次目標】<這個 session 要達成什麼，可驗收>
```

規則：
- 快照 = **指標與結論**，不是原文。環境細節引用 G，不在此展開（省 token，見 J 第 6 章）。
- `【回滾狀態】` 有內容代表上個 session 中斷在變更途中 → 本 session 第一件事是確認回滾點仍有效（文件 H）。

---

## 第 2 章：跨 session／壓縮前落地清單

在 `/compact` 前、或預期要結束 session 前，AI 必須確認以下已寫入落地檔案（非只在對話裡）：

- [ ] 中途結論 → 寫入任務單或 `CLAUDE.md`
- [ ] 若動過環境現狀（版本、port、chat_id…）→ 更新文件 G
- [ ] 未完成變更的回滾指令 → 寫入任務單 `【回滾狀態】`
- [ ] 下個 session 的開場快照 → 產出並交付使用者

未落地就結束 = 交接失敗，下個 session 會重探、燒 token（漏點 3、4）。

---

## 第 3 章：續用 vs 開新 session（決策樹）

```
context 是否逼近上限？ ─是→ 先跑第 2 章落地清單 → 開新 session，貼開場快照
                      └否↓
任務是否已切換主題／換角色？ ─是→ 開新 session（避免舊上下文污染判斷）
                            └否↓
agent loop 是否已達文件 C 清單 4 的停止門檻？ ─是→ 開新 session 重整
                                             └否→ 續用同 session
```

原則：**寧可多開乾淨 session，不要在污染的長 context 裡硬撐**——後者判斷品質下降且 token 更貴。

---

## 第 4 章：手機端最小上下文（對治 skill 不同步）

已知限制：Claude.ai web 設定的 skill 在手機 App 不可見。手機端接續時，開場快照額外內嵌所需 skill 的**關鍵片段**（非全文）：

- 需要 `ubuntu-server-setup` 時：只貼本次會用到的那一段設定（並套用文件 M 已更正的安全預設值，不要貼含 `bind 0.0.0.0` 的舊版）。
- 長 skill → 先在 web 端請 S 產一份「本任務用得到的片段摘要」，再貼到手機 session。
- 長期解法（awareness）：關鍵 skill 內容也維護一份在專案 `CLAUDE.md`／專案自訂指示，讓兩端一致（見文件 L）。

---

## 維護方式

- 與 A~M 同 repo，單一真相：直接編輯、不建副本。
- 開場快照欄位若實測有遺漏 → 改第 1 章模板，更新版本行。
- 發現新的跨裝置／壓縮限制 → 補進第 3、4 章。
