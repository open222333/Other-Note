# 文件 H｜變更管理與回滾規範（CHANGE_AND_ROLLBACK）

> 版本：1.0（2026-07-04）
> 核心鐵律：**寫不出回滾步驟的變更，不准執行。**
> 適用：所有會動到正式環境（文件 F 第 3 章的「正式-核心」「正式-冗餘」）的變更。開發機變更不需走本流程，但仍受文件 F 約束。

## 第 2 章：標準順序（固定套路，不重新發明）

### 2.1 資料庫相關變更（schema、參數、版本）
1. **先 Slave 後 Master**：Slave 上執行並觀察 ≥ 1 個驗證週期，再動 Master。
2. Master 上的 DDL 屬正式核心寫入 → 一律 ⛔（文件 F 矩陣）。
3. 變更參數優先用可動態調整的 `SET GLOBAL`（記得同步寫入設定檔），能不重啟就不重啟。

### 2.2 應用部署（Docker Compose）
1. feature 分支 → 開發機驗證 → PR → R 審查（文件 D 模板 4）→ 合併（⛔）。
2. 正式機：`git pull` → `docker compose up -d --no-deps <service>` 逐服務更新，**不用** `down` 全停再起。
3. 驗證清單全過才收工；任一項不過 → 依回滾決策點直接回滾。

### 2.3 設定檔 / .env 變更
1. 先備份（第 1 章快照）。
2. 改完 diff 給出對照，僅重啟受影響服務。
3. 同步更新文件 G 的欄位清單（若欄位有增減）。

## 第 3 章：備份與還原演練

- 備份不等於可還原。**每 90 天**至少一次還原演練：取最近備份在非正式環境還原並驗證資料可查。
- 演練結果記入文件 G 第 7 節；逾期未演練 → S 在例行檢查中自動開 D 任務單提醒。
- 備份檔的刪除/覆蓋屬文件 F 第 1 章永久禁止項，AI 只可新增與驗證。

## 第 4 章：變更凍結

以下期間禁止非緊急正式環境變更（緊急 = 文件 E 的 P1/P2 處置）：
- 還原演練進行中。
- 已知外部服務異常期間（如 Bitbucket、Linode 事故）。
- 使用者宣告的凍結窗（如出貨旺季）：<待填>

## 第 5 章：與其他文件的關係

- 變更中出事 → 切換到文件 E 劇本，事故優先於變更完成。
- 變更完成的定義 = 文件 C 清單 2 全過 **+** 本單驗證清單全過 **+** 文件 G 已同步。
