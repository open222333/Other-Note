# 文件 B｜模型調度守則（MODEL_DISPATCH）

> 版本：1.0（2026-07-04）
> 前提：遵守文件 A 第 0 章制度總則。
> 原則：**用最便宜、剛好夠用的模型完成任務；只有在明確觸發升級條件時才往上派。**

---

## 第 1 章：角色分工表

| 角色代號 | 模型層級 | 負責任務 | 明確不負責 |
|---------|---------|---------|-----------|
| **S（Scout，搜尋兵）** | 輕量（Haiku 級） | 檔案檢索、log 掃描、grep 定位、格式轉換、依文件 C 清單打勾判斷、產生任務摘要 | 修改任何程式碼、做架構判斷 |
| **W（Worker，實作手）** | 中量（Sonnet 級） | 依文件 D 模板實作與重構、寫測試、修 bug、寫設定檔、日常研究整理 | 未經模板交辦的自由發揮、跨系統架構決策 |
| **R（Reviewer，審查官）** | 重量（Opus/Fable 級） | Code review、架構決策、資安審視、疑難排解（W 兩次失敗後接手）、制度文件修訂 | 例行瑣事（浪費成本） |

> 對應到實務：Claude Code 中以 `/model` 或子 agent 設定切換；API/Telegram bot 中以 model 參數指定。

---

## 第 2 章：預設派工路由

```
新任務進來
 ├─ 只需「找/讀/列/轉格式」→ S
 ├─ 需要「寫/改/建」且有 D 模板 → W
 ├─ 需要「判斷對不對/該不該」→ R
 └─ 不確定 → 先派 S 做任務摘要，依摘要中的任務類型重新路由
```

## 第 3 章：升級標準（何時往上派）

由執行中的模型自行對照，**符合任一條即升級**，不得硬撐：

**S → W**
- 任務需要修改任何檔案。
- 需要跨兩個以上檔案推理因果關係。
- 文件 C 的判斷清單出現任何一項「無法確定」。

**W → R**
- 同一問題嘗試 2 次仍失敗（含測試不過、部署失敗）。
- 修改會影響：資料庫 schema、replication 設定、認證機制、對外網路暴露面。
- 需要在兩個以上可行方案間做取捨。
- 發現本制度文件（A~D）本身有誤。

**升級時必附**：下方第 5 章的「交接摘要」，禁止轉貼完整對話（痛點一）。

## 第 4 章：降級標準（何時往下派）

- R 完成決策後，剩餘為機械性執行 → 產出 D 模板交給 W。
- W 完成實作後，剩餘為驗證與整理（跑 checklist、掃 log、更新索引）→ 交給 S。
- 任何模型發現當前步驟「照清單打勾即可完成」→ 一律降給 S。

## 第 5 章：統一回報格式

所有角色完成或交接任務時，輸出以下區塊（缺欄位視為未完成）：

```markdown
### 回報
- 任務：<一句話>
- 角色：S / W / R
- 狀態：完成 ✅ / 升級 ⬆ / 降級 ⬇ / 阻塞 ⛔（需問使用者）
- 動過的檔案：<清單，無則寫「無」>
- 驗證方式與結果：<指令 + 輸出重點>
- 剩餘風險：<無則寫「無」>
- 下一步（交接時必填）：<給下一個角色的最小上下文：目標、相關檔案、已排除的做法>
```

## 第 6 章：成本護欄

- S 單次任務讀取上限：10 個檔案或 2000 行；超過先回報再繼續。
- W 單次任務修改上限：5 個檔案；超過拆單（回到文件 D 重新交辦）。
- R 只在第 3 章條件觸發時使用；連續兩件事都派 R，需自問是否路由錯誤。
