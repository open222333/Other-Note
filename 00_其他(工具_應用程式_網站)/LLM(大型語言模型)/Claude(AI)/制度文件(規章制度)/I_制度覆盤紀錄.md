# 文件 I｜制度覆盤紀錄（RETROSPECTIVE_LOG）

> 版本：1.0（2026-07-04）
> 定位：讓制度自我進化的機制。每則紀錄**固定三行**，多寫視為違規（防止本檔本身變成 token 黑洞）。
> 新紀錄一律加在「§2 紀錄」**最上方**（新到舊）。本檔只增不刪，是全套文件中唯一允許無限累積的檔案；但超過 50 則時，由 S 將最舊 25 則濃縮為 §3 的統計行後刪除原文。

---

## §1 觸發時機與寫法

**必須寫一則的時機：**
1. 任何 P1/P2 事故解決後（文件 E）。
2. 任何任務被退回重做（回報格式缺欄、驗收不過）。
3. AI 觸發文件 F 的 ⛔/❌、或文件 C 清單 4 的停止重試。
4. 制度文件 A~H 任何一次修訂。
5. 每週例行（週五）：本週若無上述事件，寫一則「本週無事件」亦可。

**固定三行格式：**

```
[YYYY-MM-DD][類別:事故/退件/攔截/修訂/例行] 發生什麼：<一句話>
制度哪裡沒接住 / 哪裡接住了：<指向具體文件與章節，如 E劇本3、C清單2第4項>
採取的修改：<改了哪個檔案哪一行 / 無需修改>
```

**類別判讀給 S 的規則：**
- 有走文件 E 劇本 → 事故
- 回報被打回 → 退件
- ⛔/❌/停止重試被觸發且成功防止問題 → 攔截（這是制度**成功**的紀錄，同樣要寫）
- 改了 A~H → 修訂

## §4 每月檢視問題（使用者 + R，每月一次，10 分鐘）

1. 「攔截」是否為零？為零通常代表清單沒被真的執行，而不是沒有風險。
2. 同一根因出現 ≥ 2 次 → 它應該從紀錄升級成文件 A 的痛點或 E 的劇本。
3. 有沒有哪條制度規則從未被用到？連續兩個月未用 → 考慮刪除（制度也要瘦身）。
4. 退件率是否下降？沒有下降 → 檢討 D 模板是否欄位不清，而不是檢討執行的模型。
