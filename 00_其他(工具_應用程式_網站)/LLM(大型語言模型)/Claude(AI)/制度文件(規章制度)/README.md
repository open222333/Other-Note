# README｜AI 協作制度文件使用說明

> 版本：2.0（2026-07-04）新增 E~I 五份文件
> 本目錄是 AI 協作制度的**單一事實來源**。修正一律直接編輯原檔，禁止建立副本；錯誤內容直接刪除，歷史追溯交給 Git。

---

## 檔案總覽

### 核心四件（怎麼派工、怎麼判斷）

| 檔案 | 定位 | 誰在什麼時候讀 |
|------|------|---------------|
| `A_自動化診斷檔.md` | 制度總則 + 三大痛點與修復方向 + 舊筆記待修正清單 | 人：規劃修復工作時。AI：接到與痛點相關的任務時 |
| `B_模型調度守則.md` | S/W/R 角色分工、派工路由、升降級標準、統一回報格式、成本護欄 | 人：交辦前決定派給誰。AI：每次執行任務全程遵守 |
| `C_判斷力驗證清單.md` | 四張打勾清單：何時升級、何時算完成、何時問人、何時停止重試 | AI（尤其輕量模型）：每個小步驟結束時逐項打勾 |
| `D_任務交辦模板.md` | 實作 / 重構 / 研究 / 審查四種填空模板（變更/部署模板見 H） | 人：交辦任何任務前複製填空 |

### 安全與運維四件（出事怎麼辦、什麼不准碰）

| 檔案 | 定位 | 誰在什麼時候讀 |
|------|------|---------------|
| `E_事故處置手冊.md` | 五套固定劇本：replication 斷裂、Redis、Telegram、磁碟滿、Compose 起不來 + 事故分級 | AI/人：**告警當下第一個打開的檔案** |
| `F_權限與危險操作邊界.md` | 永久禁止清單 + 角色×操作權限矩陣 + 主機分級 + ⛔ 提問格式 | AI：**執行任何指令前**先對照；與 C 衝突時以 F 為準 |
| `G_環境事實檔.md` | 主機、版本、port、.env 欄位、金鑰盤點、備份現況（含 `<待填>` 欄位） | AI：接任務先讀，禁止重新探索環境；環境有變動的任務結案前必須同步更新 |
| `H_變更管理與回滾規範.md` | 變更單模板（含強制回滾欄）、先 Slave 後 Master 等標準順序、還原演練 | 人/AI：任何要動正式環境的變更，先開 H 變更單 |

### 進化機制

| 檔案 | 定位 | 誰在什麼時候讀 |
|------|------|---------------|
| `I_制度覆盤紀錄.md` | 三行式覆盤日誌（事故/退件/攔截/修訂）+ 每月檢視問題 | AI：觸發時機發生時寫入。人：每月花 10 分鐘看 §4 |

## 文件間關係（一張圖看懂）

```
                 ┌── 平時 ──────────────────────────────┐
交辦任務 → D 填模板 → B 路由派工(S/W/R) → 執行中每步跑 C
                │            │
                │      指令執行前對照 F（不准碰的直接擋）
                │      環境資訊一律查 G（不重新探索）
                │
     要動正式環境？→ 先開 H 變更單（沒有回滾步驟＝無效）
                │
     完成 = C清單2全過 + 回報符合B格式 (+H驗證全過 +G已同步)
                 └──────────────────────────────────────┘

                 ┌── 出事 ──────────────────────────────┐
告警 → E 通用開場 → 對應劇本 → 卡住升R / 觸發⛔問人(F格式)
                 └──────────────────────────────────────┘

任何事故/退件/攔截/修訂 → I 寫三行覆盤 → 每月檢視回頭改 A~H
```

## 快速上手：交辦一個任務的標準流程

```
1. 判斷任務類型（實作/重構/研究/審查/變更）
2. 複製模板填空：前四種用 D，變更/部署用 H 第 1 章
   （關鍵欄位：「相關檔案（只准讀這些）」＋「驗收標準」＋ H 的「回滾步驟」）
3. 依 B 第 2 章路由派給 S / W / R（不確定 → 先派 S 做摘要）
4. AI 執行中：指令前對照 F 矩陣；每小步跑 C 清單 1 與 4；環境資訊查 G
5. 回報「完成」前：C 清單 2 全部打勾（變更類另加 H 驗證清單）
6. 收報告：不符 B 第 5 章格式 = 未完成，退回（並記 I 一則「退件」）
7. 推翻舊筆記 → 當場刪修；環境有變 → 當場更新 G
```

## 給 AI 的最小指令範例

```
你是角色 <S/W/R>，遵守以下制度：
- F_權限與危險操作邊界.md（硬底線，優先級最高）
- B_模型調度守則.md（回報格式、升降級標準）
- C_判斷力驗證清單.md（每步驟打勾）
- 環境事實一律引用 G_環境事實檔.md，不要自行探索
以下是任務單：
<貼上填好的 D 或 H 模板>
```

> Claude Code 使用者：將本目錄放入專案並在 `CLAUDE.md` 引用；或設定 S/W/R 為子 agent，system prompt 各自引用 B、C、F。事故用途可另建一個只掛 E+F+G 的精簡 agent，事故時 token 最省。

## 四條鐵律（違反任何一條即視為任務失敗）

1. **F 的 ❌ 永遠不做、⛔ 永遠先問**——包含使用者口頭同意也不解鎖第 1 章禁止清單。
2. **只讀任務單列出的檔案，環境資訊只查 G**——控 token。
3. **完成 = 通過 C 清單 2 全部項目**——不接受「理論上會過」；正式環境變更另需 H 驗證全過。
4. **推翻舊觀念必須當場刪修舊筆記、環境變動必須當場更新 G**——不留錯誤內容、不建副本。

## 導入順序建議

1. 立即生效：F（權限底線）、E（事故劇本）——這兩份不依賴其他準備。
2. 第一週：補齊 G 的 `<待填>` 欄位（可開一張 D 研究單派給 S+W 完成）。
3. 開始強制：所有交辦走 D/H 模板、回報走 B 格式。
4. 第一個月底：跑一次 I §4 的每月檢視，依實際退件與攔截情況修 A~H。

## 維護方式

- 全套文件放入私有 Git repo（建議 `docs/ai-governance/`）。
- 制度有誤 → 依 B 第 3 章升 R 修訂，更新該檔開頭版本行，並在 I 記一則「修訂」。
- 新痛點 → A；新劇本 → E；新任務型 → D/H；新判斷情境 → C；新環境事實 → G。
