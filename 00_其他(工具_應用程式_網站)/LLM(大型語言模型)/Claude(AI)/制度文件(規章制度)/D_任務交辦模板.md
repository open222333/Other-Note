# 文件 D｜任務交辦模板（TASK_TEMPLATES）

> 版本：1.0（2026-07-04）
> 使用方式：複製對應模板、填空、整段貼給執行模型。`<>` 為必填，`[]` 為選填。
> 共同規則：**「相關檔案」以外的內容不需要讀**；完成前必跑文件 C 清單 2；回報用文件 B 第 5 章格式。

---

## 模板 1：實作（新功能 / 新腳本）— 派給 W

```markdown
# 任務類型：實作
# 角色：W（Sonnet 級）

## 目標（一句話）
<要做出什麼，做給誰用>

## 背景（≤3 行）
<為什麼要做；只寫影響實作的事實>

## 相關檔案（只准讀這些）
- <path/to/file1>：<這個檔案跟任務的關係>
- [path/to/file2]

## 規格
- 輸入：<>
- 輸出：<>
- 錯誤處理：<例如：Telegram 400 需處理 migrate_to_chat_id>
- 禁止事項：[例如：不得硬編碼 chat_id / 不得引入新套件]

## 驗收標準（每條都要能被指令驗證）
1. <指令或測試> → <預期結果>
2. <>

## 環境
- 主機：<Master 64GB / Slave 32GB / 本機>
- 分支：<feature/xxx>（不得直接改 main）
```

## 模板 2：重構（不改行為，只改結構）— 派給 W

```markdown
# 任務類型：重構
# 角色：W（Sonnet 級）

## 目標（一句話）
<把什麼改成什麼樣的結構>

## 不變式（重構後必須完全相同的行為）
1. <對外介面 / 輸出格式 / 副作用>
2. <>

## 相關檔案（只准讀這些）
- <>

## 重構範圍
- 允許改動：<>
- 禁止改動：<例如：public API 簽名、設定檔格式>
- 修改檔案上限：5 個（超過先回報拆單）

## 驗收標準
1. 重構前先執行 <既有測試/驗證指令> 並記錄輸出。
2. 重構後同一指令輸出一致。
3. `git diff --stat` 只含允許範圍內的檔案。
```

## 模板 3：研究（查資料 / 比較方案）— 派給 S，必要時升 W

```markdown
# 任務類型：研究
# 角色：S（Haiku 級）

## 問題（一句話）
<要回答什麼>

## 判斷用途
<這個答案會拿來決定什麼——讓研究知道該多深>

## 範圍
- 需要涵蓋：<例如：MySQL 8.4 的行為，不是 8.0>
- 明確排除：<不用比較的選項>
- 資料來源優先序：專案內筆記 → 官方文件 → 其他

## 產出格式（固定）
1. 結論（≤3 行，先講答案）
2. 依據（條列，每條附出處）
3. 對本專案的影響：<需要改哪個檔/哪份筆記；若推翻舊筆記 → 列出待刪修段落>
4. 不確定之處（無則寫「無」）

## 禁止
- 禁止在研究任務中修改任何檔案。
```

## 模板 4：審查（Code review / 設定審視）— 派給 R

```markdown
# 任務類型：審查
# 角色：R（Opus/Fable 級）

## 審查對象
- <PR 連結 / diff / 檔案路徑>
- 變更摘要（交辦人填，≤3 行）：<>

## 審查重點（依序）
1. 正確性：<這次特別擔心的點>
2. 安全性：認證資訊是否硬編碼？對外暴露面是否擴大？（Redis bind / 防火牆 / port）
3. 一致性：是否違反制度文件 A~D？是否與 ubuntu-server-setup 筆記衝突？
4. [效能：<例如 buffer pool / 慢查詢影響>]

## 產出格式（固定）
- 結論：通過 ✅ / 修正後通過 🔧 / 退回 ❌
- 問題清單：每項含「位置、問題、建議修法、嚴重度(高/中/低)」
- 高嚴重度為 0 才可標「通過」
- 若發現舊筆記/制度文件需要修正 → 直接列出應刪除與應替換的段落
```
