# 文件 C｜判斷力驗證清單（JUDGMENT_CHECKLIST）

> 版本：1.0（2026-07-04）
> 使用方式：輕量模型（角色 S）逐項打勾。**規則只有一條：任何一項打不了勾，就執行該清單指定的動作，不自行推理繞過。**

---

## 清單 1：何時該升級任務（每完成一小步就跑一次）

全部為「否」才可以繼續留在目前角色；任一「是」→ 依文件 B 第 3 章升級。

- [ ] 這一步需要修改檔案，而我是 S？（是 → 升 W）
- [ ] 我已經對同一個錯誤做了第 2 次修復嘗試？（是 → 升 R）
- [ ] 這次修改會碰到：MySQL replication 設定 / 資料庫 schema / SSH 或 API 認證 / 防火牆與對外 port？（是 → 升 R）
- [ ] 我在兩個做法之間猶豫，而任務單沒有指定？（是 → 升 R）
- [ ] 我發現制度文件 A~D 的內容和現實矛盾？（是 → 升 R）

## 清單 2：何時算完成（回報「完成 ✅」前必跑）

**全部打勾才算完成**，缺一項狀態只能填「阻塞」或繼續做：

- [ ] 任務單（文件 D 模板）中的「驗收標準」每一條都有對應的驗證輸出。
- [ ] 驗證是**實際執行**的結果（指令輸出 / 測試通過 / curl 回應），不是「理論上會過」。
- [ ] 動過的檔案清單與 `git status` 一致，沒有意外多改的檔案。
- [ ] 沒有留下暫時性寫死的值（chat_id、IP、密碼、路徑）；有的話已移入 `.env` 或設定檔。
- [ ] 若本任務推翻了舊筆記/舊文件的內容 → 已直接刪修該筆記（不是新增副本）。
- [ ] 回報格式（文件 B 第 5 章）每個欄位都有填。

## 清單 3：何時需要問使用者（狀態改為「阻塞 ⛔」並停手）

任一「是」→ 停止動作，輸出問題，等待回覆：

- [ ] 動作不可逆或難以復原？（刪除資料、`DROP`、`RESET`、覆蓋備份、rotate 金鑰）
- [ ] 需要在**正式環境的 Master 資料庫**上執行寫入類操作？
- [ ] 會造成服務中斷或重啟？（mysqld / redis / docker compose restart 於正式環境）
- [ ] 任務單缺少必要資訊，且猜錯的代價高？（例如不確定目標主機是 Master 還是 Slave）
- [ ] 需要花費金錢或變更外部帳號設定？（開新服務、改 DNS、改 Bitbucket 權限）
- [ ] 任務要求與制度文件衝突，且清單 1 升級後 R 也無法在制度內解決？

> 問問題的格式：一次只問一題，附上「我建議的預設選項」，讓使用者可以只回「好」。

## 清單 4：何時停止重試（防止 token 燒錢迴圈）

- [ ] 同一指令/同一修法已失敗 2 次 → 停止，升 R。
- [ ] 錯誤訊息連續 2 輪完全相同 → 停止，升 R。
- [ ] 已讀取超過 10 個檔案仍找不到目標 → 停止，回報目前排除範圍，請求縮小範圍。
- [ ] 單一任務對話已需要我回頭翻找開頭的約定 → 停止，輸出交接摘要，建議開新 session。
