# 文件 M｜Skill 維護政策（SKILL_MAINTENANCE）

> 版本：1.0（2026-07-07）
> 定位：把 skill 檔納入單一真相治理。定義 skill 的審查／更正節奏、安全預設值底線、與新增／修改的檢查點。分工原則：**步驟留 skill，現狀進 G（文件 G），行為約束進 CLAUDE.md（文件 L）**。
> 執行方式：新增或修改任何 skill 前過第 3 章檢查點；既有 skill 依第 4 章節奏審查；安全設定一律對照第 2 章。

---

## 第 1 章：待更正項（已確認，須直接修正 `ubuntu-server-setup`）

`ubuntu-server-setup` skill 內含不安全的 Redis 設定，屬**直接更正**項（單一真相：覆寫，不加註解、不留舊值）：

| 現況（錯） | 應改為 | 理由 |
|-----------|--------|------|
| `bind 0.0.0.0` | `bind 127.0.0.1`（或內網固定 IP） | 不對公網開放 6379（呼應文件 E 劇本 2 的攻擊風險） |
| `protected-mode no` | `protected-mode yes` | 保留 Redis 內建保護 |
| 明文密碼寫在設定檔 | `requirepass` 取用強密碼，值由部署時注入、**不寫進 skill** | 金鑰不落 skill（文件 F 禁令） |

更正後同步：Redis 現行 `bind`／`requirepass` 的**現狀**記到文件 G，不留在 skill。

---

## 第 2 章：安全預設值底線（所有 skill 通用）

skill 產出的任何設定範例，不得違反以下底線（違反即等同文件 F 禁令，審查一律退回）：

- 不對公網綁定內部服務（DB、Redis、內部 API）——預設 localhost 或內網。
- 不關閉安全機制當作解法：`StrictHostKeyChecking no`、關防火牆、`protected-mode no` 一律禁止寫進 skill。
- 不在 skill 內出現任何金鑰／token／密碼明文——一律 `$ENV_VAR` 或「部署時注入」。
- MySQL 8.4 認證：不使用已移除的 `mysql_native_password`（呼應已知外部狀態腐化項）。

---

## 第 3 章：新增／修改 Skill 檢查點

改任何 skill 前逐項打勾，任一不過即停：

- [ ] 這是「步驟／方法」嗎？（是 → 放 skill；若是「現狀事實」→ 改放 G）
- [ ] 有無金鑰明文？（有 → 改 `$ENV_VAR`）
- [ ] 設定範例是否過第 2 章底線？
- [ ] 是否與既有 skill 重複？（重複 → 合併，不建第二份）
- [ ] 描述（description）觸發條件是否精準？（避免誤觸／漏觸，浪費 token）
- [ ] 手機端會用到嗎？（會 → 依文件 K 第 4 章備妥可內嵌片段）

---

## 第 4 章：審查節奏

- **事件驅動**：每次某 skill 被實際使用且發現內容過時／有誤 → 當場開更正，改完更新該 skill 開頭版本行。
- **定期**：配合文件 G 的易腐清單核對週期，一併掃 skill 是否引用了已變的外部狀態（版本、認證方式、host key 假設）。
- **退役**：不再使用的 skill 直接刪除，不保留「以防萬一」的死檔（單一真相）。

---

## 維護方式

- 本政策與 A~M 同 `docs/ai-governance/`；skill 本體在其各自位置版控。
- 第 1 章待更正項完成後，把該列標記為「已更正 <日期>」或直接移除該列（依單一真相，建議移除並在 Git commit 記錄）。
- 新發現的安全底線 → 補進第 2 章，更新版本行。
